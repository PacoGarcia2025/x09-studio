7. Banco Supabase

20 tabelas públicas com RLS. RPCs financeiras apenas com service_role.

GrupoTabelas
Identidadeprofiles, workspaces, projects
Pipeline Nextplans, plan_tasks, plan_task_logs, verify_reports, fix_runs
Visual MVPvisual_projects
Billingbilling_plans, subscriptions, credit_wallets, credit_ledger, usage_logs, billing_webhook_events
GitHubgithub_installations, github_repositories, github_oauth_states, github_webhook_deliveries
Deployproject_deployments

Regras de crédito

  • Signup → wallet + 5 créditos
  • Build/edit → 1 crédito
  • Plan / repair → 0
  • Pacotes: Basic 100 / Pro 500
  • RPCs: debit_generation_credits, grant_credit_package, apply_ledger_entry
Dois mundos de projeto: projects (pipeline Next + disco) e visual_projects (MVP + JSON) são modelos separados. O produto visual atual usa só visual_projects.

9. Riscos e pendências

  1. Aplicar migrations SaaS no Supabase remoto (o deploy da VPS não roda SQL).
  2. Configurar GitHub App e Vercel na VPS (health ainda false).
  3. Configurar webhook Mercado Pago no painel para creditar automaticamente.
  4. Middleware de /api/projects/** olha cookie; cliente Vite só com Bearer pode ser redirecionado.
  5. phase=plan no orquestrador pode executar build sem débito (divergência billing vs agent).
  6. Débito de crédito ocorre antes do LLM; não há refund automático se a geração falhar.
  7. /api/publish ainda não publica de verdade.
  8. Checkout atual é compra avulsa; tabelas/cancel ainda têm vestígios de assinatura.
  9. Itens do catálogo /ai e /ecosystem (Stripe, Resend, R2, Sentry…) são só planejamento.
  10. Deploy PM2 com delete + start gera janela de downtime.
